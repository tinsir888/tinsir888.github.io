The Existence of Core

Is the core empty?

In the paper, we have proved that Greedy Capture computes 222-core in O(kn)O(kn)O(kn) time for maximum distance loss.

❓Is the 222 the lower bound for code?

Special Case: Binary Distance Value

Consider a graph G=(V,E)G=(V,E)G=(V,E), where for every pair i,j∈Vi,j\in Vi,j∈V, (i,j)∈E(i,j)\in E(i,j)∈E if dist(i,j)=1dist(i,j)=1dist(i,j)=1, else dist(i,j)=2dist(i,j)=2dist(i,j)=2.

If there exist a graph such that no matter how we assign the clusters, there always exist a clique of size at least ⌈n/k⌉\lceil n/k\rceil⌈n/k⌉, then the lower bound of 222 is proved.

Split&Merge Algorithm

In any given aforementioned graph GGG, the cluster in the core always exist. (But this can not prove the actual core always exist.)

Consider the algorithm below:

Algorithm Split-and-Merge:

Input: Graph G=(V,E)G=(V,E)G=(V,E), kkk

Output: Clustering C=(C1,⋯ ,Ck)C=(C_1,\cdots,C_k)C=(C1​,⋯,Ck​) in the actual core

  1. n←#V,C←∅,X←∅n\gets\#V,C\gets\empty,X\gets\emptyn←#V,C←∅,X←∅
  2. while X≠VX\neq VX=V, do // SPLIT PHASE
    • C′←MAXCLIQUE(V╲X,E)C'\gets MAXCLIQUE(V\diagdown X,E)C′←MAXCLIQUE(V╲X,E)
    • C←C∪{C′}C\gets C\cup\{C'\}C←C∪{C′}
    • X←X∪C′X\gets X\cup C'X←X∪C′
  3. while ∣C∣>k|C|\gt k∣C∣>k, do // MERGE PHASE
    • c1←arg⁡min⁡c∈C∣c∣,c2←arg⁡min⁡c∈C╲{c1}∣c∣,c′=c1∪c2c_1\gets\arg\min_{c\in C}|c|,c_2\gets\arg\min_{c\in C\diagdown\{c_1\}}|c|,c'=c_1\cup c_2c1​←argminc∈C​∣c∣,c2​←argminc∈C╲{c1​}​∣c∣,c′=c1​∪c2​
    • C←C╲{c1,c2},C←C∪{c′}C\gets C\diagdown\{c_1,c_2\},C\gets C\cup \{c'\}C←C╲{c1​,c2​},C←C∪{c′}
  4. return CCC

Analysis of Split&Merge

It’s straightforward to see that Split&Merge is a greedy algorithm.

Since Split&Merge uses MAXCLIQUEMAXCLIQUEMAXCLIQUE as an oracle, it is a NP solution.

🎯Theorem: Split-and-Merge always return the cluster in the actual core.

Claim: If a set of points form a cluster, which means, the cost of everybody is 111. All of them have no incentive to form a coalition with anyone outside of the clique.

Thus before merging, any subset of VVV that intersects at least two item-sets in CCC, can not be a clique.

❗By pigeonhole principle, before merging, any item-set in CCC with size already greater than ⌈n/k⌉\lceil n/k\rceil⌈n/k⌉ would not be picked in the merge phase! All data points in these clusters have no incentive to deviate.

In the merge phase, every point who would have been merged have the cost 222. But they can not find other points to form a clique with size at least ⌈n/k⌉\lceil n/k\rceil⌈n/k⌉.

If there exist a clique with size at least ⌈n/k⌉\lceil n/k\rceil⌈n/k⌉ in the merge phase, by pigeonhole principle and the property of maximum clique, it should be chosen in the split phase. Which leads to a contradiction.

🤔Can we extend Split&Merge to more general cases?

Special Case: Trinary Distance Value

Consider a group of points NNN, for every pair of points, their distance can only be 1,2,21,\sqrt2,21,2​,2. (Now matter how the distances are, they always subject to triangle inequality)

If there exists an instance, where no matter how we assign kkk clusters, the approximation of core cannot be better than 2\sqrt22​, then we prove that the bound for inapproximability of core is 2\sqrt22​.

It seems that we can extend Split&Merge in this case to produce clustering in the core.

Extend Split&Merge for Trinary Distance Value

The extended version of Split&Merge:

Input: kkk and the distance between each pair of points

  2. X←V,C←∅X\gets V,C\gets\emptyX←V,C←∅

  4. while ∣c∣≥⌈n/k⌉|c|\ge\lceil n/k\rceil∣c∣≥⌈n/k⌉ do // find the clique of distance 111 with large enough size
    • C←C∪{c}C\gets C\cup\{c\}C←C∪{c}
    • X←X╲cX\gets X\diagdown cX←X╲c

Analysis Split&Merge for Trinary Case

For simplicity, we introduce the notation of core-violator: A core-violator is a subset S⊆NS\subseteq NS⊆N with size ≥⌈n/k⌉\ge\lceil n/k\rceil≥⌈n/k⌉, where each point in SSS has a strictly better cost if they are clustered together.

🤔Claim: extended version of Split&Merge always return a clustering in the core.

Assume that Split&Merge returns a clustering not in the core. Which means there exist a core-violator SSS.

  • Since the minimal distance is 111 and the cost of each points in the clique in CCC is exactly 111 before Split&Merge, core-violator can’t exist there.

  • The only possible existence of core-violator is in the Split&Merge step.

    • Claim: It is not possible that the a free point at step 5 can collude with any point in the clique formed in step 4.
    • The only possible core-violator can formed only among free points at step 5.
    • ❗️By the property of step 4, any subset of free points at step 5 can not form a core-violator with distance 111 between any pair. Otherwise they they should be clustered in step 4.
      • Thus, the possible core-violator must contain a free point improving cost from 222 to 2\sqrt22​, meaning that there exist a large enough clique in G′G'G′ after the merge phase.
      • By pigeonhole principle, this contradicts the property of split phase (This large enough clique must be chosen in split phase).

Thus the extended version of Split&Merge guarantees the core property.

❓Can we further extend Split&Merge to general cases where there are n(n−1)/2n(n-1)/2n(n−1)/2 distances?

Extend Split&Merge to General Cases

For nnn data points in a metric space, given number of clusters kkk.

We construct the complete weighted undirected graph G=(V,E)G=(V,E)G=(V,E) where VVV corresponds to the set of data points, for each pair (i,j)∈E(i,j)\in E(i,j)∈E, the weight is the distance between iii and jjj. There are n(n−1)/2n(n-1)/2n(n−1)/2 edges in GGG.

First, we sort the edges by weight.

Start from another graph with VVV and no edge. We iteratively add an edge from the scratch in the ascending order.

  • At each time, we check whether there exist a clique of size at least ⌈n/k⌉\lceil n/k\rceil⌈n/k⌉.
    • If so, we immediately include them into a cluster and delete them temporarily.

🤔Intuitively I think this extension outputs a cluster in the core. But I haven’t well organized a formal proof.
